Исправь баг: $ARGUMENTS

## Методология

### 1. Исследование
- Прочитай описание проблемы из $ARGUMENTS
- Если указан номер GitHub issue, получи детали: `gh issue view <номер>`
- Определи, какие файлы и подсистемы затронуты
- Воспроизведи проблему мысленно, проследи flow данных по архитектуре

### 2. Диагностика (Root Cause Analysis)
- Найди точное место, где возникает баг
- Определи корневую причину (не симптом!)
- Проверь, есть ли аналогичные проблемы в других местах кода
- Задокументируй цепочку: входные данные → что должно произойти → что происходит фактически

### 3. Планирование исправления
- Опиши план исправления перед написанием кода
- Рассмотри краевые случаи: пустой ввод, отсутствие токенов, сетевые ошибки, таймауты
- Убедись, что исправление не ломает другие подсистемы (calendar/openclaw — независимы)
- Если PostgreSQL не подключён — бот должен продолжать работать

### 4. Исправление
- Внеси минимальные точечные изменения
- Добавь проверки и guard clauses для предотвращения повторения
- Если нужна валидация входных данных — добавь на границе (вход от Telegram, ответ от API)
- Сохрани стиль: ES Modules, `.js` imports, двойные кавычки, strict TypeScript

### 5. Self-review
- Проверь исправление как строгий ревьювер
- Убедись: type safety, обработка ошибок, нет регрессий
- Проверь: `npx tsc --noEmit` проходит без ошибок
- Запусти: `npm run build` для проверки сборки

### 6. Отчёт
Выведи:
- **Корневая причина** — одно предложение
- **Что исправлено** — список изменённых файлов с описанием
- **Защита от повторения** — какие проверки добавлены
- **Затронутые подсистемы** — calendar / openclaw / voice / db / api
