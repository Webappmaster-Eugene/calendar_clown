# Инструкция по использованию и как это устроено

## Как пользоваться ботом

### Запуск диалога

1. Найдите своего бота в Telegram (по имени, которое вы задали в @BotFather).
2. Нажмите **Start** или отправьте команду `/start` — бот ответит приветствием и списком команд.

### Создать встречу текстом

- Отправьте команду вида:  
  **`/new Встреча с командой завтра в 15:00`**
- Бот разберёт фразу (дата, время, название), создаст событие в вашем Google Calendar и пришлёт подтверждение со ссылкой на календарь.
- Можно писать по-русски: «завтра в 15:00», «в понедельник 10:00», «встреча с Иваном 12 марта в 14:00» и т.п.

### Создать встречу голосом

- Отправьте **голосовое сообщение** (иконка микрофона в поле ввода).
- Скажите одной фразой, что за встреча и когда, например: «Встреча с бухгалтерией завтра в десять утра».
- Бот ответит «Обрабатываю голосовое…», затем распознает речь, поймёт событие и создаст его в календаре (или напишет, что не удалось разобрать).

### Отправить сообщение кому-то голосом (только доверенные)

- Доверенные пользователи (см. `ADMIN_USER_IDS`) могут голосом попросить бота отправить сообщение другому пользователю. Например: «Отправь Анжелике Надточеевой что я ее люблю».
- Получатель определяется по **имени и фамилии** или по **username** — среди тех, кто уже хотя бы раз писал боту (бот хранит привязку в `data/user_chats.json`).

### Посмотреть встречи

- **`/today`** — список встреч на сегодня.
- **`/week`** — встречи на текущую неделю.
- **`/list`** — то же, что `/today`.

### Справка

- **`/help`** — краткий список команд в чате.

## Как это устроено под капотом

Этот бот — **отдельное приложение** (Node.js, Telegraf). Оно не является частью OpenClaw и не вызывает OpenClaw при обработке сообщений. Ниже — что реально происходит при каждом действии.

### Общая схема

```
Вы в Telegram
      │
      ▼
 Telegram API (получение сообщений)
      │
      ▼
 Наш бот (один процесс на VDS)
      │
      ├── Текст (/new, /today, /week) ──► разбор дат (chrono-node) или список событий
      │                                        │
      │                                        ▼
      │                                  Google Calendar API (OAuth2)
      │
      └── Голосовое сообщение ──► 1) Скачивание OGG
                                   2) Транскрипция (OpenRouter, модель с аудио)
                                   3) Извлечение события (OpenRouter, DeepSeek)
                                   4) Google Calendar API (создание события)
```

- **OpenClaw** на VDS может работать параллельно (свой бот, свои каналы), но в цепочке этого календарного бота он **не участвует**: ни текст, ни голос через OpenClaw не идут.

### Текстовые команды (/new, /today, /week)

1. Бот получает сообщение из Telegram.
2. **`/new ...`** — из текста после `/new` библиотека **chrono-node** (с поддержкой русского) извлекает дату, время и название события. Бот вызывает **Google Calendar API** (ваш календарь по OAuth2, токен в `data/token.json`) и создаёт событие.
3. **`/today`** и **`/week`** — бот запрашивает у Google Calendar список событий за выбранный период и форматирует ответ в чат.

Никакой нейросети для текстовых команд нет — только парсер дат и API календаря.

### Голосовые сообщения (цепочка «под капотом»)

1. **Telegram** отдаёт боту голосовое сообщение (файл в формате OGG).
2. Бот **скачивает** этот файл во временную папку.
3. **Транскрипция (STT):** OGG в base64 отправляется в **OpenRouter** (модель с поддержкой аудио, например GPT Audio Mini). Возвращается текст фразы.
4. **Намерение и извлечение:** текст отправляется в **OpenRouter** (модель **DeepSeek Chat v3.1**) одним запросом. Модель определяет намерение и возвращает JSON:
   - **Календарь** — одно событие (название, начало, конец в ISO, Europe/Moscow). Бот создаёт событие в Google Calendar и пишет подтверждение.
   - **Отправить сообщение** — получатель (имя/фамилия или username) и текст. Доступно только доверенным пользователям (`ADMIN_USER_IDS`). Бот ищет chat_id по хранилищу `user_chats` (по имени или username) и отправляет сообщение от имени бота. Получатель должен уже писать боту ранее.
   - **Иначе** — ответ «не удалось понять».
5. Временный голосовой файл удаляется.

Итого для голоса: **Telegram → наш бот → OpenRouter (транскрипция + один вызов на намерение и поля) → Google Calendar или отправка сообщения**. OpenClaw в этой цепочке не используется.

### Хранение сообщений в PostgreSQL (опционально)

Если задана переменная `DATABASE_URL` (или `POSTGRES_*`), бот сохраняет все входящие сообщения (текст и голосовые) и после обработки голоса — транскрипт и тип намерения (`calendar` / `send_message` / `unknown`) в таблицу `messages`. Поднять БД: `docker compose up -d`, применить миграции: `npm run migrate`. Без БД бот работает как раньше, запись просто не выполняется.

### Где может быть OpenClaw

- **OpenClaw** — это отдельный шлюз для своих ботов и каналов (WhatsApp, Telegram, Discord и т.д.), со своими агентами и инструментами. Его можно держать на том же VDS.
- **Этот календарный бот** — отдельный процесс (например, systemd-сервис `telegram-calendar-bot`). Он знает только: Telegram API, OpenRouter и Google Calendar. С OpenClaw он не связан.
- Если вы захотите в будущем «перевести» голос или календарь в OpenClaw, это будет уже другая интеграция (например, свой инструмент внутри OpenClaw, вызывающий ваш бот или API календаря).

### Кратко

| Действие           | Что реально вызывается                          |
|--------------------|--------------------------------------------------|
| `/new` текст       | chrono-node + Google Calendar API                |
| `/today`, `/week`  | Google Calendar API                              |
| Голосовое          | ffmpeg (OGG→WAV) → OpenRouter (транскрипция + DeepSeek) → Google Calendar API |

## Роль OpenClaw в данном пайплайне

### OpenClaw в этом приложении не используется

В пайплайне **календарного бота** (текст, голос → Google Calendar) **OpenClaw не участвует**. Ни один запрос от пользователя не идёт в OpenClaw и не обрабатывается им.

Цепочка полностью замкнута на:

- **Telegram API** — приём и отправка сообщений;
- **наш процесс** (Node.js, Telegraf) — логика бота;
- **OpenRouter** — транскрипция голоса (модель с аудио) и извлечение события из текста (DeepSeek);
- **Google Calendar API** — создание и чтение событий.

Переменные окружения `OPENCLAW_GATEWAY_URL` и `OPENCLAW_GATEWAY_TOKEN` в этом проекте **не используются**. Конфигурация OpenClaw на VDS на работу бота не влияет.

### Что такое OpenClaw и зачем он мог бы быть

**OpenClaw** — это отдельный шлюз (gateway) для подключения мессенджеров (Telegram, WhatsApp, Discord и т.д.) к AI-агентам. Он:

- принимает сообщения из каналов по своему боту/настройкам;
- может распознавать голос (встроенная транскрипция);
- держит сессии, агентов, инструменты (tools) и маршрутизацию;
- отдаёт ответы обратно в мессенджер.

То есть OpenClaw — это **другое приложение**: свой Telegram (или иной) бот, свои агенты, свой конфиг. Его можно ставить на тот же VDS и использовать параллельно с календарным ботом для других сценариев (общий ассистент, другие команды и т.п.).

### Почему в названии репозитория есть «OpenClaw»

Изначально ставилась задача «общаться с ним [OpenClaw] через Telegram и чтобы он выполнял команды», в том числе запись в календарь. В итоге был выбран вариант **отдельного календарного бота** с узкой задачей (только календарь) и прямыми вызовами OpenRouter и Google Calendar — без прохождения через OpenClaw. Так проще деплой, один ключ OpenRouter и один чёткий контекст (календарь). Название репозитория сохранило отсылку к окружению (VDS с OpenClaw), но сам OpenClaw в коде и в пайплайне не фигурирует.

### Если нужно задействовать OpenClaw

Чтобы календарь реально шёл через OpenClaw, потребовалось бы:

1. Настроить в OpenClaw **свой** Telegram-бот (или общий бот с маршрутизацией).
2. Добавить в OpenClaw **инструмент** (tool), например «создать встречу в календаре», который по HTTP вызывает ваш backend или напрямую Google Calendar API.
3. Научить агента OpenClaw вызывать этот инструмент по голосу или тексту.

Тогда пайплайн выглядел бы так: **Telegram → OpenClaw → агент решает вызвать инструмент «календарь» → ваш сервис или API создаёт событие**. В текущей реализации вместо этого используется **Telegram → наш бот → OpenRouter + Google Calendar** без OpenClaw.

## API отправки сообщений пользователям по username (для OpenClaw)

OpenClaw (или другой доверенный сервис на том же VDS) может **отправлять сообщения пользователям бота по имени пользователя (username)**. Сообщение уходит в личку от имени этого же бота.

### Зачем в этой схеме нужен OpenClaw

**OpenClaw** — шлюз с агентами и инструментами: он принимает входящие сообщения из мессенджеров (свой Telegram/WhatsApp/Discord и т.д.), обрабатывает их через AI-агентов и может вызывать внешние инструменты (tools). При этом агенту часто нужно **инициировать исходящее сообщение** конкретному пользователю Telegram — по username, а не по chat_id.

Через Telegram Bot API **нельзя** получить chat_id по username: такого метода нет. Поэтому агент OpenClaw не может сам «достать» chat_id и отправить сообщение в личку. Зато **этот календарный бот** уже общается с пользователями в Telegram и при каждом обращении сохраняет привязку username → chat_id. Он же владеет одним Telegram-ботом и может от его имени слать сообщения в известные чаты.

В схеме получается так: **OpenClaw (агент) решает, кому и что написать** — и вызывает наш HTTP API: «отправь пользователю @username такой-то текст». **Наш бот** по сохранённой привязке находит chat_id и отправляет сообщение от имени подключённого бота. То есть OpenClaw задаёт «кому и что», а этот бот выполняет доставку в Telegram тем пользователям, у которых уже есть диалог с ботом.

### Как это работает

1. При любом обращении пользователя к боту в личке бот сохраняет привязку **username → chat_id** в файл `data/user_chats.json` (или путь из `USER_CHATS_PATH`). Telegram не даёт получить chat_id по username через API, поэтому без такого предварительного контакта отправить сообщение по username нельзя.
2. Если задана переменная `SEND_MESSAGE_API_KEY`, при запуске поднимается HTTP API на порту `SEND_MESSAGE_API_PORT` (по умолчанию 18790), хост — `SEND_MESSAGE_API_HOST` (по умолчанию 127.0.0.1).
3. OpenClaw вызывает **POST /send** с телом JSON и заголовком авторизации.

### Формат запроса

- **URL:** `POST http://127.0.0.1:18790/send` (порт и хост из env).
- **Заголовок:** `Authorization: Bearer <SEND_MESSAGE_API_KEY>`.
- **Тело (JSON):** `{ "username": "johndoe", "text": "Текст сообщения" }`.
  - `username` — Telegram username получателя (без `@` или с ним).
  - `text` — текст сообщения от имени бота.

### Ответы

- **200:** `{ "ok": true }` — сообщение отправлено.
- **400:** неверный JSON или нет поля `username`.
- **401:** нет или неверный Bearer token.
- **404:** пользователь не найден (никогда не писал боту или у него нет username).
- **500:** ошибка Telegram при отправке (в теле есть `error` и при необходимости `details`).

### Переменные окружения

| Переменная | Описание |
|------------|----------|
| `SEND_MESSAGE_API_KEY` | Секрет для Bearer; если не задан, API не запускается. |
| `SEND_MESSAGE_API_PORT` | Порт (по умолчанию 18790). |
| `SEND_MESSAGE_API_HOST` | Хост (по умолчанию 127.0.0.1). |
| `USER_CHATS_PATH` | Путь к файлу хранилища (по умолчанию `./data/user_chats.json`). |
